物联网平台技术方案

这是一个面向工业设备接入、数据采集、状态管理和业务控制的物联网平台项目， 采用 Go + EMQX + Kafka + TDengine + PostgreSQL 的组合，支撑设备接入、消息流转、 时序存储、命令下发和多租户管理等核心能力。

技术栈：Go / MQTT / EMQX / Kafka / TDengine / PostgreSQL 架构风格：少拆服务、强边界、先可用、可演进 多租户：逻辑隔离优先 可靠性：默认 at-least-once

1. 总览与目标

第一版的重点不是把平台能力一次做满，而是先把物联网最关键的业务闭环打稳： 设备接入、身份鉴权、上行遥测、消息解耦、时序落库、查询、下行控制、ACK 回执、 在线状态和基础告警。

推荐范围：设备预注册、设备接入鉴权、MQTT 上行遥测、Kafka 解耦、TDengine 入库与查询、 下行命令与 ACK、在线状态查询、基础告警。

目标

  • 打通设备接入、消息传输、时序入库和查询闭环。
  • 保证 MQTT / Kafka / TDengine 链路具备重试、幂等和 DLQ 能力。
  • 支持下行命令、ACK 和命令状态机。
  • 支持逻辑多租户隔离和基础权限控制。

非目标

  • 第一版不做复杂规则引擎。
  • 第一版不做完整设备影子。
  • 第一版不做多地域双活。
  • 第一版不引入 Schema Registry / 配置中心。
主链路接入 → 解耦 → 入库 → 查询
可靠性at-least-once + 幂等
隔离tenantId 贯穿全链路
演进先少拆服务，后按需扩展

2. 总体架构

平台的核心链路可以理解为：设备通过 MQTT 接入 EMQX，接入服务做鉴权和标准化， 将事件写入 Kafka，再由 worker 异步落 TDengine 和 PostgreSQL；管理后台和外部系统通过 REST API 查询元数据、命令状态、在线状态和历史数据。

graph TD D[设备] -->|MQTT PUBLISH| E[EMQX] E -->|消息回调/订阅| I[Go ingress] I -->|标准事件| K[Kafka] K --> W[Go worker] W --> T[TDengine] W --> P[PostgreSQL] U[管理后台/外部系统] --> A[Go admin] A --> P A --> T I -->|命令下发| E E -->|ACK/状态| I

推荐组件

  • EMQX：MQTT 接入、Topic ACL、会话保持、离线消息。
  • Kafka：接入解耦、削峰、重试、DLQ、重放。
  • TDengine：时序写入、时序查询、聚合分析。
  • PostgreSQL：业务元数据、命令、告警、权限、当前态。

架构原则

  • 第一版优先简单，服务数量尽量少。
  • 端到端默认按 at-least-once 设计。
  • 所有写路径都预留幂等能力。
  • 所有数据都携带 tenantId。

3. 核心数据流

这一部分按业务场景拆成子标签页，方便直接查看不同链路的职责、顺序和状态变化。

上行遥测步骤

  • 设备发布 MQTT 到上行 Topic。
  • EMQX 完成连接和 ACL 判断。
  • ingress 校验设备身份、版本和字段。
  • 标准事件写入 Kafka。
  • worker 消费后落 TDengine，并同步关键状态到 PostgreSQL。

上行时序图

sequenceDiagram participant D as Device participant E as EMQX participant I as Go ingress participant K as Kafka participant W as Go worker participant T as TDengine participant P as PostgreSQL D->>E: PUBLISH /up E->>I: 回调/订阅消息 I->>I: 鉴权 / 校验 / 标准化 I->>K: produce telemetry event K->>W: consume event W->>T: insert timeseries W->>P: upsert latest state W-->>K: commit offset

下行命令步骤

  • 管理后台创建命令，admin 写 PostgreSQL 并生成 commandId。
  • ingress 按租户与设备上下文发布下行消息。
  • 设备收到命令后返回 ACK。
  • 命令状态更新为 pending / dispatched / sent / acked / timeout / failed。

命令时序图

sequenceDiagram participant U as 管理后台 participant A as Go admin participant P as PostgreSQL participant I as Go ingress participant E as EMQX participant D as Device U->>A: POST /commands A->>P: insert command(pending) A-->>U: commandId A->>I: dispatch request I->>E: publish /down E->>D: deliver command D->>E: publish /ack E->>I: ack callback I->>P: update command status

告警步骤

  • 遥测进入后先做轻量阈值判断。
  • 触发后将告警写 Kafka 或 PostgreSQL。
  • admin 提供告警列表、详情和通知消费入口。

告警时序图

sequenceDiagram participant D as Device participant I as Go ingress participant K as Kafka participant W as Go worker participant P as PostgreSQL D->>I: telemetry I->>I: threshold / N次连续判断 alt 触发告警 I->>K: alarm event K->>W: consume alarm W->>P: store alarm / notify end

在线状态判定

  • 采用 EMQX 连接状态 + 设备心跳 的双来源判定方式。
  • 业务库保存当前态即可，不把在线状态当作单一强一致真相。
  • 核心字段以最后活跃时间和会话状态为主。

常用字段

字段用途
last_seen_at最后上报或心跳时间
connected当前在线标识
session_id连接会话标识
heartbeat_timeout_sec在线超时阈值

4. 协议与 Topic

Topic 设计建议采用统一前缀 tenant/{tenantId}/device/{deviceId}/...。 这样在 EMQX ACL、日志检索、问题排查、设备隔离和命令投递上都更清晰。

用途 Topic 示例 说明
上行遥测 tenant/{tenantId}/device/{deviceId}/up 设备发往平台的标准数据
下行命令 tenant/{tenantId}/device/{deviceId}/down 平台发往设备的控制命令
ACK 回执 tenant/{tenantId}/device/{deviceId}/ack 设备对命令或事件的确认
事件通知 tenant/{tenantId}/device/{deviceId}/event 设备异常、状态变化、补充事件
在线状态 tenant/{tenantId}/device/{deviceId}/status 在线、离线、心跳、会话状态

消息模型

第一版建议统一使用 JSON，不要先上 Protobuf。原因很直接：设备联调、日志排查、回放排错和协议演进都更容易。 消息体必须包含固定头部和业务 payload，版本字段用于兼容演进。

必备字段

  • msgId
  • tenantId
  • deviceId
  • ts
  • type
  • version
  • payload

建议字段

  • traceId
  • productId
  • region
  • seq
  • schemaVersion
{
  "msgId": "b7a0d8c8-4f8b-4b1e-9d7d-3ad4d7fe1d2a",
  "tenantId": "t1",
  "deviceId": "d1",
  "ts": 1717670000000,
  "type": "telemetry",
  "version": "v1",
  "traceId": "trace-001",
  "payload": {
    "temp": 23.4,
    "humi": 60.1
  }
}
版本兼容策略建议定为：向后兼容优先。新增字段只增不删，接入服务负责兼容旧版本，不要让设备升级绑定后端发布节奏。

5. 存储设计

存储设计也拆成子标签页，分别说明 Kafka、TDengine、PostgreSQL 和幂等去重策略。

Kafka 设计

  • Topic 按业务域拆，不按设备拆。
  • 推荐：iot.telemetry、iot.event、iot.command、iot.dlq。
  • 分区键优先 deviceId，保证同设备有序。
  • 失败重试后进入 DLQ，原始消息和失败原因要保留。

语义约束

  • 默认接受 at-least-once。
  • 消费端必须幂等。
  • 支持重试、DLQ 和人工重放。
  • 保留 traceId，便于全链路追踪。

TDengine 建模

  • 采用超级表 + 标签。
  • 标签建议：tenantId、deviceId、productId、region、deviceType。
  • 固定列放高频稳定字段，扩展字段放 ext JSON。
  • 原始数据建议保留 30 到 90 天，聚合数据保留更久。

建模原则

  • 高频稳定字段用固定列。
  • 扩展差异字段用 JSON。
  • 原始数据和聚合数据分开存。
  • 避免把所有点位做成超宽动态表。

PostgreSQL 表域

  • 承载租户、用户、角色、设备、命令、告警和状态数据。
  • 适合复杂事务、条件查询和状态流转。
  • 建议统一作为业务元数据主库。

核心表域建议

  • tenant / user / role / permission
  • product / device / device_secret / device_cert
  • command / command_ack / alarm
  • device_state / message_dedup / rule_config

幂等策略

  • 以 msgId + deviceId + ts 组合进行去重。
  • 消费端和入库端都要支持幂等写入。
  • 重复消息不应引发重复告警或重复命令状态变更。

建议实现

  • Kafka 消费前先查去重表或缓存。
  • 写入时用唯一键约束兜底。
  • 失败原因进入 DLQ，支持人工重放。

6. 安全与多租户

权限控制建议采用两层设计：EMQX 负责设备接入鉴权和 Topic ACL，查询管理服务负责租户、用户、角色和设备权限。 第一版用逻辑多租户隔离即可，不需要物理隔离。

设备侧

  • 支持预注册和激活流程。
  • 第一版优先设备 ID + Secret，条件成熟后可升级到证书方案。
  • 设备上线必须绑定租户和产品上下文。

租户侧

  • 所有记录必须带 tenantId。
  • 查询接口必须强制按租户过滤。
  • Topic、日志、Kafka 消息都应携带租户标识。
  • 第一版采用逻辑隔离，不做物理隔离。
边界 负责内容 备注
EMQX 设备能否连上来、能发哪些 Topic、能订阅哪些 Topic 适合做接入侧第一道防线
业务服务 用户能看哪些设备、能否下发命令、能否管理租户 按租户上下文进行鉴权

7. 运维与可观测性

推荐第一版采用单地域单集群，业务服务无状态化，依托 Kubernetes 做部署和扩缩容。 网关只负责薄层鉴权、限流、路由，不承载复杂业务逻辑。

可观测性

  • 日志要结构化输出，包含 tenantId、deviceId、msgId、traceId。
  • 指标建议覆盖接入成功率、Kafka 堆积、TDengine 入库失败率、在线数。
  • 链路追踪至少覆盖 MQTT 接收 → Kafka → 入库。

部署建议

  • Go 服务做无状态部署。
  • EMQX、Kafka、TDengine、PostgreSQL 按各自集群能力扩容。
  • 配置先用 ConfigMap / Secret + 环境变量。
  • 不优先引入独立配置中心。

9. 交付路线

第一版的正确姿势不是把所有平台能力做完，而是先把主链路做稳，再一点一点补齐可扩展能力。

阶段 交付内容 验收关注点
Phase 1 设备预注册、MQTT 鉴权、上行遥测、Kafka、TDengine 链路可用、消息不丢、写入可追踪
Phase 2 下行命令、ACK、命令状态机、在线状态、基础告警 控制闭环完成，状态一致性可解释
Phase 3 DLQ、重放、聚合报表、设备影子、规则增强 平台具备可运维和可扩展能力
交付节奏建议：先把“设备接入、消息可靠流转、时序存储、命令闭环”跑稳，再进入平台化和规模化演进。

10. 风险与决策

主要风险

  • 设备侧协议不统一，导致接入层兼容复杂。
  • 消息重复和乱序引发告警、命令和状态错判。
  • 过早拆分服务导致排障成本升高。
  • 时序与业务元数据边界不清，后期查询和建模返工。

已定决策

  • 设备直连 EMQX。
  • Go 订阅 MQTT 后写 Kafka。
  • Kafka topic 按业务域拆，分区键用 deviceId。
  • TDengine 采用超级表 + 标签。
  • 业务库选 PostgreSQL。
  • 第一版采用逻辑多租户隔离。
规则判断建议保持轻量，避免第一版引入独立规则引擎；先把协议、幂等和状态闭环做扎实，后面再扩展编排和分析能力。
不建议第一版就做：复杂事件溯源、Schema Registry、多地域双活、过度拆分的微服务矩阵。
